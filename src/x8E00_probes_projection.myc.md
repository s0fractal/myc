# Myc Probes Projection

Draft bridge per `SUBSTRATE_SELF_ABI.v0.1` slot 8/E. Mirrors the omega and
liquid projection patterns.

**Myc has no `probes/` directory** in the trinity sense. Trinity uses `probes/`
for try-it-and-see experimental dirs that may graduate to live `src/` organs.
Myc's experimental edges live differently:

- as **`protocols/<area>/SPEC.draft.md`** files — design drafts being iterated
  before formal Phase activation
- as **`public/objects/`** — content-addressed thoughts captured but not yet
  projected into named transformations or recipes

This projection names where myc's experimental frontier lives, so trinity's
federated probes view can include myc honestly instead of reporting "myc has 0
probes" (which would be wrong).

## Flow

```text
myc/protocols/<area>/SPEC.draft.md (design checkpoint drafts)
myc/public/objects/ (raw captured thoughts, not yet projected)
  -> myc/src/x8E00_probes_projection.myc.md (this — frontier bridge)
  -> trinity federated probes view
```

## Experimental Frontier Signals

### Protocol Drafts (formal experimental edges)

Each `myc/protocols/<area>/SPEC.draft.md` represents an experimental edge: a
pattern being designed before formal Phase activation. Current areas (as of
2026-05-23):

- **capabilities** — Phase 5 design checkpoint; payload-bearing capability
  descriptors not yet enabled
- **consensus** — Phase 9 witness/trust topology drafts
- **jazz** — Phase 7 JAZZ integration formalization
- **publishing** — Phase 8 PublishDescriptor + multi-witness consensus
- **recipes** — Phase 4 recipe layer (mostly graduated; remaining drafts test
  edge cases)
- **resolver** — schema drafts for verify-projections-response, etc.
- **sealed** — sealed receipt drafts (Phase 5)

For per-area status, see each `protocols/<area>/SPEC.draft.md`.

### Content-Addressed Frontier (informal experimental surface)

`public/objects/h/<short>/` accumulates raw captured thoughts that have a
commitment hash but no transformation-graph projection yet. Until a
`TransformationDescriptor` references them, they're experimental in the same way
an unrun probe is — possible-future crystallization material.

Per myc design (MYC.md): `naming: computed-after-capture` — names emerge from
content, so the experimental surface is dynamic rather than
directory-structured.

### Why No Literal probes/ Dir

Myc's content model is content-addressed, not coordinate-addressed like trinity.
A `probes/` dir would impose path-addressed thinking on a content-addressed
substrate. Per `myc/MYC.md`: `raw_resolution: "descriptor-first"`.

Experimental frontier is a property of the descriptor's lifecycle (draft →
enabled), not of its filesystem location.

## Projection Contract

Myc owns this projection. Trinity should consume this file as myc's declared
experimental-frontier signal. Trinity should NOT mine `protocols/` or
`public/objects/` directly to enumerate probes; that would treat myc's
content-addressed surface as a coordinate-addressed one and miscount.

If myc ever DOES grow a literal `probes/` dir (e.g., for runtime behavioral
probes against the resolver API), this projection should be retired in favor of
the same scan pattern trinity uses for its own `probes/`.

This projection is hand-authored (claude-opus-4-7 2026-05-23). Future direction
is for it to be partially generated by scanning `protocols/*/SPEC.draft.md`
files and emitting their frontmatter, similar to how trinity's `t probes` reads
probe `SPEC.md` banners.
